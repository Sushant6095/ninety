# ADR 054 Terminal SSOT × halt money-shot reconciliation — match STATUS is single-source via the store, C2's halt choreography is kept whole

Status: accepted 2026-07-14 (implemented on `merge/live-integration`; main untouched). Extends: ADR-050 (item 3.2 — "elevate the halt into the full River-integrated money-shot"). Cross-ref: ADR-051 (WC26 two-source rule — the live/static split this sits alongside), ADR-005/028 (goal-halt-reanchor + halt/reopen hardening — the domain behaviour the money-shot dramatizes), ADR-042/048 (home data layer + fixture-wired deploy — the seam the open follow-up plugs into).

Context: a 3-track parallel build ran in git worktrees. All three worktree agents branched from the STALE session-start HEAD (`8293076`) rather than the post-commit HEAD (`c91f843`), so Track 1's terminal SSOT refactor was built WITHOUT C2's already-committed GSAP halt choreography. Merging Track 1 produced a 4-file architecture clash on the terminal — `MatchColumn`, `MatchHeader`, `BigRiver`, `useTerminalLive`. Two designs collided: **C2's Tier-0 halt money-shot** (the demo cold open — `useHaltSequence` GSAP timeline, `BoothLine`, the `useTerminalLive` reprice/settle engine, cliff-draw/sweep/wash DOM overlays, a `headerLive` override) versus **Track 1's SSOT store** (`matchLiveStore.ts` / `useMatchLive` / `setMatchStatus`), which had replaced the choreography with a static amber watermark.

Decision:
- **Keep C2's full halt money-shot AND make match STATUS the single source of truth via `matchLiveStore`.** `MatchColumn` reads `view` from `useMatchLive(TERMINAL_MATCH_ID)` and writes every transition through `setMatchStatus`; `StateSwitcher` and the halt sequence's `HaltActions` (reset/halt/resume) also write via `setMatchStatus`. So header, switcher, `MarketStatus`, and the board all agree on status from one store — the choreography no longer owns a private status flag.
- **Retain `useTerminalLive` as the terminal's choreographed price engine.** The store has no reprice/settle, and the terminal match (Australia v Egypt, `MATCH.matchId`) is NOT on the board slate — so there is no cross-surface price divergence to arbitrate. The store's `driftTick` remains the board's ambient price engine; the two engines drive disjoint surfaces by construction, not by discipline.
- **`layout.tsx` keeps both mounts:** the `CssStudio` dev mount and the `MatchLiveProvider` (wrapping `children`).

Rejected alternative: **"SSOT wins" — take Track 1's terminal files wholesale.** Rejected because it silently gutted the demo cold-open to a static amber watermark and would have left the GSAP timeline + `BoothLine` files as dead code on disk. Status-SSOT was the real win in Track 1; the watermark was collateral, not the goal.

Consequences / VERIFY: `tsc` 0 errors; `next build` green (17 pages); `main` untouched. Status flows one-way — grep confirms `MatchColumn`, `StateSwitcher`, and `HaltActions` all mutate through `setMatchStatus` and read `view` from `useMatchLive`, with no residual private status flag in the halt sequence. OPEN FOLLOW-UP (deliberately not done): the fixtures→live swap is NOT a seed swap. The web models markets as **H/D/A (match-winner)**; the live API (Track 2) currently serves **OU/AH (over/under)** books. Wiring fixtures→live requires either confirming the API supports a WINNER market kind (only OU/AH was seeded) or adding an H/D/A↔over/under adapter, verified against a running stack — so `NEXT_PUBLIC_USE_FIXTURES` was intentionally NOT added this session. When resolved, that swap gets its own ADR.

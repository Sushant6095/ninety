# ADR 018 ingest normalization — odds.raw (raw per-market), match.events (derived), fixtures compacted, idempotency

Status: accepted 2026-07-07. Context: worker-ingest (L1) subscribes the odds + scores SSE via `packages/txline`, normalizes to canonical Envelopes (`packages/schema`), and publishes to the bus (`odds.raw`, `match.events`) with `(source, source_seq)` idempotency and a `match_id` partition key; fixtures are polled. Probing the live devnet feed (ADR-015/016) forced several concrete decisions that reality, not the map, dictated. Decision (verified against the live devnet feed 2026-07-07): (1) `odds.raw` carries the RAW per-market odds tick as a LOOSE Envelope — `type "odds_tick"` is a category tag only, `payload` = the raw fields `{fixtureId, ts, bookmaker, superOddsType, marketParameters, priceNames, prices, pct}`. Reality: the live devnet odds stream emits per-bookmaker markets (only `OVERUNDER_PARTICIPANT_GOALS` was observed — no 1X2 / match-result market appeared), so the strict `OddsTickPayload {outcomes:{H,D,A}}` schema does NOT match the raw feed. `odds.raw` is therefore a raw pass-through; cortex derives the 1X2 fair prices downstream (`prices.marks`). This continues the ADR-015 "reality wins" thread — the schema's 1X2 `OddsTickPayload` is a mismatch to reconcile in a schema session (add a raw-odds payload, or redefine `odds_tick`); ingest publishes via the bus's loose Envelope so nothing is blocked. Rule: raw ingest streams are loosely typed; strictly-typed events are the PROCESSED streams (`match.events` goals, `prices.marks` Mark). (2) Idempotency `(source, source_seq)`: scores `source_seq` = `ScoreState.Seq` (a real monotonic per-fixture counter); odds carry NO numeric seq, so `source_seq` = a deterministic djb2 hash of `MessageId` (same `MessageId` → same seq → a redelivered tick dedupes). Dedup is a bounded per-process Set (`Dedup` class); multi-instance / restart-durable dedup needs a shared Redis SET or a monotonic per-`(source,match)` watermark (the noted upgrade path). Partition key = `match_id` = `String(FixtureId)`. (3) Scores → `match.events` by DIFFING consecutive `ScoreState` per fixture: a goal event is emitted when `Score.*.Total.Goals` increases (team = the side that rose, minute from `Clock.Seconds`, score from Totals). `GameState`→kickoff/ht/ft mapping is DEFERRED because devnet `GameState` is unreliable ("scheduled" with a running clock — ADR-015); goals are the event that matters (goal → HALT). (4) Fixtures poll → COMPACTED upsert into a Redis hash `fixtures:current` (field = fixtureId, value = latest json), change-detected (HSET only when the record changed). This is a compacted reference store (latest-per-fixture), distinct from the append-only event streams — it fits the "hot state lives in Redis" model and is what market-creation reads; the bus law governs the odds/scores EVENT flow, which does go through `packages/bus`. (5) Structured JSON log every 100 processed ticks (`evt: ingest.progress` with counts). `main.ts` runs the REPLAY loop by default (archived `docs/txline-samples` through the pipeline → bus) — a runnable dev loop with no live auth; LIVE mode wires a chain-backed `TxLineClient` subscriber + signer (composition layer, per `scripts/txline-live.mjs`) and calls `runIngest`, kept OUT of worker-ingest to honor "only chain builds Solana txs". Consequences: (+) VERIFY green — an integration test vs docker Redis replays real odds ticks and `XLEN odds.raw.v1` grows by the number of unique ticks, and a duplicate injection is dropped by `(source, source_seq)` (grew=4, dropped=1). (+) the pipeline is transport-normalization only (no business logic), leaving 1X2 modeling to cortex. (−) the schema `OddsTickPayload {H,D,A}` still doesn't reflect the raw feed — a schema reconciliation is owed. (−) `match.events` depends on `ScoreState` diffs and is sparse on the devnet feed; status transitions are deferred. (−) per-process dedup only. No CLAUDE.md law or hook changed. Cross-refs ADR-007 (bus), ADR-015/016 (live data-spine reality), TXLINE-MAP §2.

# ADR 024 market lifecycle state machine — pure reducer, 7×7 guard table, event-time only (apps/api/src/engine/market.ts)

Status: accepted 2026-07-07. Context: the single-writer engine needs one authoritative, deterministic definition of a market's life. Builds on ADR-002 (LMSR — the machine deliberately holds NO pricing math; `amm.ts` stays the pure pricer) and ADR-005 (goal/red → HALT → reopen with 3× spread decaying over 60s). Decision (implemented, gate-green, engine-guardian-reviewed this session): (1) The lifecycle SCHEDULED→OPEN→LIVE⇄HALTED→RESOLVING→SETTLED|VOIDED is a PURE reducer `transition(state, cmd) → { state, effects }` — no IO, no LMSR, no wall clock. Every timestamp comes from the event payload (`cmd.at` / `Envelope.ts_source`), so replays are deterministic and single-writer-safe; state is `readonly`, spread-copied. (2) An explicit 7×7 guard table drives it. Triggers: open (kickoff−24h), kickoff, pivot (goal|red|penalty), reopen (policy), ft, settle (proof-verified), abandon (→void+refund). Each (state,trigger) cell is exactly one of: a legal transition (emits exactly one side-effect), a tolerated no-op (benign at-least-once redelivery / late feed), or ILLEGAL → a typed `IllegalTransitionError`; every illegal edge throws. (3) The machine only EMITS side-effects — open, live, halt{spreadMult}, halt_extend, reopen{spreadMult,decayMs}, resolving, settled{result,sig}, voided{refund} — the engine turns these into bus publishes / AMM re-anchor / refunds. (4) ADR-005 tie-in: pivot LIVE→HALTED emits halt at spread 3×; reopen decays 3→1 over 60s (`spreadAt`, event-time driven) and `shouldReopen` fires only once event-time clears the halt window; a pivot DURING halt (e.g. a red card) EXTENDS the halt (halt_extend, pushes the window) — it does NOT re-halt from scratch. (5) VOID refunds EXACTLY the committed credits: a pure `stake()` accumulator tracks credits staked, abandon emits voided{refund: committed}; the order gate `canAcceptOrder` is true only in OPEN/LIVE — HALTED rejects (the MARKET_HALTED law). (6) create-or-park registry (`apply`): an event for an unknown market creates it (SCHEDULED) on first sight; a command illegal for the current state is PARKED (dropped), never crashing the single-writer loop. Consequences / VERIFY: (+) 62 tests exhaustively cover illegal transition from every state, duplicate ft (no-op), goal-after-ft late feed (ignored, no re-halt), red-during-HALTED (extend not re-halt), VOID refund exact, unknown-market create-or-park. (+) engine-guardian review: all 6 core invariants PASS (single-writer, journal-then-ack shape, deterministic time-from-events, amm purity, no http/ws imports, halted-rejects-orders) and the 7×7 table verified correct; it found 3 issues, ALL FIXED with regression tests — (a) MEDIUM: settle no longer mints a placeholder result/sig, it REJECTS (parks) a settle lacking a verified result + on-chain sig, so a fake terminal SETTLED can never lock; (b) MEDIUM: `haltedAt`/`reopenAt` now use monotone-clamped event-time, so a stale/out-of-order stamp cannot shorten the halt window or skew spread decay; (c) LOW: `fromEnvelope` drops an unparseable `ts_source` (returns null) instead of stamping epoch 0. (~) DELIBERATE DECISION (engine-guardian flagged as ADR-worthy, not a bug): abandon is LEGAL from RESOLVING — a post-FT abandonment voids + refunds even though on-chain settlement may be in flight; there is NO engine-side interlock preventing a refund on a market the chain later settles. No CLAUDE.md law or hook changed: the machine ENFORCES the existing single-writer / no-http-ws / MARKET_HALTED / amm-purity laws, it does not alter them. Play-money invariant unaffected; no admin/result path added. Cross-refs ADR-005 (halt/reopen/decaying spread), ADR-002 (LMSR — pricing lives in amm.ts, not here). Follow-ups: (a) the engine loop `index.ts` references the machine API (createRegistry/apply/fromEnvelope) but the bus wiring (match.events → apply → publish effects; prices.marks → AMM re-anchor) is PENDING; (b) the RESOLVING-abandon vs on-chain-settled reconciliation interlock lives outside market.ts and is PENDING; (c) SettledPayload validation is a presence check in fromEnvelope — could tighten to a zod parse.

Extended-by: ADR-025 (2026-07-07): implements this reducer's pending runtime (follow-up (a)) — the durable journal-then-ack + per-market serialized loop with the match.events → apply → publish and prices.marks → AMM re-anchor bus wiring.

# ADR 021 finished-fixture replay through the ingest plane, control via the system plane

Extends ADR-018 (2026-07-07) + ADR-019 (2026-07-07); depends on ADR-020 (2026-07-07). Status: accepted 2026-07-07. Context: dev/demo and cortex/engine testing need a live-shaped feed without a live devnet fixture; ADR-018 stood up the ingestâ†’bus pipeline (`odds.raw`/`match.events`) and its `main.ts` already replays archived `docs/txline-samples`, but that loop is fixed-speed, un-triggerable, un-isolated, and stamps replayed data indistinguishably from live. This ADR makes replay a controllable, isolated, provenance-tagged capability. Implemented and gate-green in `apps/worker-ingest` + `apps/api`. Decision: (1) REPLAY MECHANICS (`apps/worker-ingest/src/replayer.ts` `replayItems`/`replayFixture`): walk a finished fixture's archived scores/odds buckets (`docs/txline-samples`), re-timestamp each event relative to replay start (wall-clock `now`), and publish through the SAME domain topics (`odds.raw`/`match.events`) at speed `Nx`. Only wall-clock `Ts` is rescaled; the in-game clock (`minute`) is PRESERVED â€” a goal lands at `(goalTs âˆ’ t0)/speed` after start while keeping its real match minute. (2) CONTROL PATH via the system plane (ADR-020): `POST /admin/replay {match_id, speed}` (admin token via `x-admin-token` header, CONSTANT-TIME compare; an unset token REJECTS all) publishes a `replay_request` `SysEvent`; worker-ingest consumes it off `sys.signals.v1` and runs the replay. WHY the system plane and not a direct call or a new "command" plane: the two-plane law (ADR-020) forces it â€” apiâ†’worker is inter-service comms and a replay trigger is not a domain event, so it MUST be a system signal; no third plane (YAGNI). This is the FIRST cross-service use of the ADR-020 plane. (3) ISOLATION by `match_id` (`ReplayManager`): one replay per `match_id` (a duplicate is REJECTED, not queued); concurrent DIFFERENT matches run in parallel, each with its OWN pipeline (separate dedup/`prevScore` â†’ no cross-talk); total concurrency capped at `MAX_CONCURRENT_REPLAYS=8` as a DoS guard. (4) PROVENANCE / play-money safety: replayed events are stamped `source="replay"` (the canonical `EventSource` value, previously dead code) so they are DISTINGUISHABLE from live data on the shared topics. This addressed a security-review HIGH: a replay of a currently-live fixture would otherwise be indistinguishable to the single-writer engine, which has NO admin result path. REQUIRED FOLLOW-UP: when `apps/api/src/engine` is wired to consume `match.events`, it MUST ignore/quarantine `source="replay"` events for any currently-live match, and `/admin/replay` MUST validate `match_id` against a finished-fixture allowlist â€” the distinguishability PRIMITIVE ships now; ENFORCEMENT is deferred because the engine is a stub today. (5) `scripts/replay.sh` wired (`bash scripts/replay.sh <fixture> [speed]`; `API` + `ADMIN_TOKEN` from env). Consequences: (+) VERIFY green via deterministic tests (virtual clock + capture bus): ordered events, goal at correct relative minute, `source="replay"` tag, `match_id` isolation, concurrency cap; admin route 401/202/400.

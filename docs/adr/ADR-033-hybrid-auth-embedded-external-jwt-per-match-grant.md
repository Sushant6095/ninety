# ADR 033 hybrid auth — embedded (email→custodial keypair) + external (Phantom sign-verify) → User + HS256 JWT; per-match 1,000-credit grant idempotent (apps/api/src/auth/*, prompt 25)

Status: accepted 2026-07-08. Context: ADR-006 chose hybrid auth (embedded default + optional external wallet); prompt 25 implements it plus the per-match play-money grant. Decision (implemented, gate 9/9/9, 128 api tests, security-reviewer-cleared): (1) EMBEDDED WALLET — `deriveEmbeddedWallet(email)` = `HMAC-SHA256(EMBEDDED_WALLET_SECRET, email.trim().toLowerCase())` → 32-byte seed → `nacl.sign.keyPair.fromSeed` → a deterministic ed25519 keypair; the base58 public key is the Solana address, the 64-byte secret is the EXPORT path (self-custody, and export is auth-gated to the wallet owner). Custody = the server holds `EMBEDDED_WALLET_SECRET`; it can always regenerate a user's wallet from their email. SIGNUP REQUIRES EMAIL-OWNERSHIP PROOF (a stateless OTP, `auth/otp.ts`): `/auth/embedded/start` emails a 6-digit code (HMAC-bound into a verification token — no server storage), and `/auth/embedded` issues the session only after the code verifies — so knowing an email string is NOT enough to log in as someone. (2) EXTERNAL WALLET — `verifyWalletSignature(pubkey, message, sig)` checks an ed25519 signature over a server-issued login challenge; only the key owner can produce it. (3) IDENTITY — a `User { walletPubkey @unique, authKind }` is upsert'd on login (embedded|external); the session is a MINIMAL HS256 JWT (`auth/jwt.ts`, no library) with `sub`=userId + `authKind` + `exp` — HS256 is PINNED (verify rejects any other `alg` → no alg-confusion/`none` bypass), signature compare is timing-safe, `exp` enforced. (4) MIDDLEWARE — `authFromBearer` (REST `Authorization: Bearer`) + `authFromWsToken` (WS `?token=`) resolve the caller or return null → the caller rejects (no anonymous orders). (5) GRANT — `grantMatchCredits` gives 1,000 credits per (user, match) EXACTLY once: idempotency reuses the ADR-027 ProcessedEvent inbox keyed `grant:{userId}:{matchId}` written in the SAME `$transaction` as the `CreditLedger` row (reason `match_grant`), so a replay hits the PK and no-ops; wired to first market open (`GET /markets/:matchId`, auth-gated). Consequences / VERIFY: (+) unit tests green — embedded wallet deterministic per email (distinct emails → distinct wallets) + 64-byte export key; Phantom ed25519 sig verify (right msg/key ✓, wrong ✗); JWT round-trip + tamper/expiry/alg-`none` all rejected; middleware accepts valid + rejects missing/garbage (unauthenticated WS order rejected); grant fires once per user·match (idempotent replay ✗, distinct user/match separate). Both auth kinds produce verifiable sessions (embedded + external can each trade). (~) The full "embedded + Phantom both trade a REPLAY end-to-end" needs the order-source gateway (submits OrderCmd with the authed userId to the engine) — an ADR-026 documented follow-up; the auth + grant primitives it depends on are green here. (+) SECURITY-REVIEWER (mandated for auth) found + FIXED 2 CRITICAL + 1 HIGH before ship: CRITICAL — `/auth/embedded` issued a full session to anyone who knew a victim's email (no ownership proof) → now gated by the stateless OTP above; the dev-fallback secrets had no boot enforcement → `auth/secrets.ts` now THROWS in production if JWT_SECRET/EMBEDDED_WALLET_SECRET are unset (`assertSecretsAtBoot`, called in startHttp) and domain-separates the JWT / login-challenge / OTP HMAC keys via a KDF (no shared raw key). HIGH — the per-match grant fired BEFORE checking the match existed (unlimited credit farming on bogus matchIds) → `GET /markets/:matchId` now 404s without granting if the market doesn't resolve. CONFIRMED CLEAN by the reviewer: HS256 alg-pinning is real (verification algorithm is hardcoded, not header-selected; alg:none rejected), timing-safe compares throughout, NO IDOR (userId is always server-derived from the verified JWT), grant idempotency is DB-enforced (ProcessedEvent PK). (−) Residual follow-ups (MEDIUM/LOW, recorded, play-money-weighted): rate-limit the `/auth/*` surface (`@fastify/rate-limit`) to make Sybil/farming impractical; the login challenge + OTP are exp-bounded but NOT single-use (a captured signature/code is replayable within its short TTL — add a used-nonce set if needed); the 7-day JWT has no per-token revocation (rotate JWT_SECRET to mass-invalidate); the OTP email transport is a pluggable PROD dependency — dev logs the code, prod must inject a real email `OtpSender` (the default `UnconfiguredOtpSender` throws). Cross-refs: ADR-006 (hybrid auth decision), ADR-003/027 (the credit ledger + ProcessedEvent inbox the grant reuses), ADR-026 (the order gateway that carries the authed user).

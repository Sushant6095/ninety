# ADR 031 on-chain leaderboard claim — authority-gated root, sorted-pair Merkle inclusion, receipt-PDA double-claim guard, PDA-owned SPL vault (programs/omnipitch_core, prompt 20)

Status: accepted 2026-07-08. Context: prompt 20 — post a leaderboard Merkle root per epoch (authority-gated), let a user claim their points by proving inclusion of leaf=(pubkey, amount), guard against double-claim, and pay out SPL tokens from a program vault whose mint authority is a PDA. This is the first REAL on-chain instruction built now that ADR-030 unblocked `anchor build`/`anchor test`. Decision (implemented, `anchor test` 5/5 green on localnet, `anchor build` clean, proof-auditor reviewed): (1) MERKLE (`merkle.rs`, PURE + host-tested) — leaf = `keccak256(claimer_pubkey(32) ‖ amount_le(8))`; internal nodes = `keccak256(sorted pair)` (OpenZeppelin-style). Sorted-pair folding makes proofs order-independent and compact. Binding BOTH the claimer pubkey AND the amount into the leaf means a proof for user A / amount X cannot be replayed by user B or for a different amount — `verify_proof` recomputes the root from the on-chain `claimer.key()` and the passed `amount`, so any substitution yields a different root and fails. Extracted as pure functions so the security core is host-unit-tested (`cargo test`) independent of the validator. (2) DOUBLE-CLAIM GUARD — a `ClaimReceipt` PDA seeded `["claim", epoch_le, claimer]` created with `init` inside `claim_points`; its mere existence means this `(epoch, claimer)` already claimed, so a replay hits "account already in use" and the whole tx reverts (the transfer never runs). The PDA is derived from the claimer's own key, so a caller cannot dodge it by passing a different receipt account. (3) VAULT (`init_vault.rs`, `claim_points.rs`) — a single PDA `["vault_auth"]` is BOTH the points-mint authority AND the vault token-account owner; the program is therefore the SOLE minter (points can't enter circulation except via this program), and claim payouts are an SPL `transfer` from the `["vault"]` token account signed by the `vault_auth` PDA seeds. Mint + vault live at PDAs `["mint"]`/`["vault"]` so they're unambiguous and can't be substituted; `claimer_ata` is constrained `token::mint = mint, token::authority = claimer` so funds can only land in the caller's own account for the right mint. `init_vault` mints the initial claim supply into the vault. (4) AUTHORITY GATE — a `Config` PDA `["config"]` holds the authority (set once by `init_config`); `post_leaderboard_root` is `has_one = authority`, so only the configured authority may post. Each epoch's `PointsEpoch` PDA `["points", epoch_le]` is created with `init`, so a root is write-once per epoch — it cannot be silently overwritten (a re-post of the same epoch fails). NO admin claim/settle bypass exists (programs/CLAUDE.md law). (5) The "vault init script" is `scripts/init-vault.ts` (calls `init_config` + `init_vault`); the same flow is exercised by the anchor test's `before()`. Consequences / VERIFY: (+) `anchor test` 5/5 on localnet — valid claim credits the exact amount ONCE; a replay FAILS (receipt PDA); a WRONG proof FAILS; a FOREIGN/tampered leaf (amount not in the tree) FAILS; `post_leaderboard_root` rejects a non-authority signer. Plus a host `cargo test` for the pure Merkle logic (valid / wrong-proof / foreign-leaf / tampered-amount / single-leaf edge) and a clean `anchor build`. (~) Toolchain: `anchor-lang` 0.30.1 with anchor-cli 0.32.1 (a version-mismatch WARNING — build + test pass; pinning the CLI via avm is deferred). Play-money invariant intact — points are a program-minted SPL token with no cash path. (+) PROOF-AUDITOR: no CRITICAL/HIGH — all four attack surfaces confirmed closed (merkle forgery/replay: the leaf is recomputed from the on-chain `claimer.key()` + `amount`, both bound, so no cross-user or cross-amount replay; leaf preimage 40B vs node preimage 64B are domain-separated; double-claim: the receipt PDA is a pure function of (epoch, claimer) with no `close` path; vault: all of vault/mint/authority are seed-derived PDAs + `claimer_ata` is mint/owner-constrained so funds can't be redirected or over-drawn; authority: `has_one` + write-once `init` per epoch). (−) Follow-ups (proof-auditor MEDIUM/LOW + scope): INIT_CONFIG FRONT-RUN (MEDIUM) — `init_config` is permissionless, so on a fresh deploy the first caller wins `authority`; MITIGATED in practice by an ATOMIC deploy+init (scripts/init-vault.ts runs both back-to-back). The proper hardening is an upgrade-authority gate on `init_config` — implemented + reverted this session because `anchor test`'s localnet does not deploy the program upgradeable-with-a-matching-authority, so the gate can't pass in the test harness; it lands once deploy provisioning provides a known upgrade authority. (LOW) `PointsEpoch.total` is unused (reserved for a future per-epoch payout-budget cap; today the only ceiling is the vault supply); (LOW) `claim_points` accepts `amount == 0` (harmless self-inflicted rent burn). Plus: the off-chain leaderboard producer (the settlement saga, prompt 24) must build the SAME sorted-pair keccak tree and hand each winner their proof; a claim window / epoch-close and reclaim-of-unclaimed-supply policy; the anchor CLI/lang version alignment (0.32.1 vs 0.30.1). Cross-refs: ADR-030 (the build unblock that made this testable), ADR-003 (the off-chain points ledger this on-chain claim pays out against), ADR-017/029 (the prior blocker), programs/CLAUDE.md (permissionless, proof-gated, no admin path).

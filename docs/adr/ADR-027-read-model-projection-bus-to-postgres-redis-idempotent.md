# ADR 027 read-model projection — engine effects → Postgres + Redis hot state, idempotent by event_id (apps/api/src/services/projection*.ts · engine/emit.ts · packages/schema credits.v1)

Status: accepted 2026-07-08. Context: the engine (ADR-025/026) computes fill/position/ledger effects, but its bus-emit adapter was a stub ("topic mapping TBD") and no consumer wrote them to the read model — so a filled trade was durable in the journal yet invisible in any user's balance/positions. Decision (implemented, gate-green): (1) ENGINE EMIT — the `startEngine` adapter now publishes effects as bus Envelopes via a PURE mapper (`engine/emit.ts`): a `fill` fans out to an order row (orders.v1, status "filled") + a fill row (fills.v1) joined by a deterministic `order_id = ${marketId}:${n}` (one order command → one order); `position` → positions.v1 (avg = the paired fill price, a v1 cost-basis approximation); `ledger` → credits.v1; `settled` → settlement.v1. Each Envelope's `event_id` is a DETERMINISTIC uuid5-style sha256 of `engine:${marketId}:${n}:${k}`, so a re-emit of the same effect carries the same id and downstream dedup is idempotent. `emit` was widened to `(marketId, n, at, effects)` (n = the journal seq = the id source; at = event-time for ts_source). New schema (packages/schema, additive): `credits.v1` topic + `CreditEvent`/`CreditPayload`. `reject` effects and non-`settled` lifecycle effects are intentionally NOT routed here (follow-ups). (2) PROJECTION CONSUMER (`services/projection*.ts`) — subscribes orders/fills/positions/credits (consumer group "projection", its own cursor → replays retained streams) and writes Postgres (Order/Fill/Position/CreditLedger) + Redis hot state (`pos:{u}:{m}:{o}`, `bal:{u}`, `lb:global` zset, `credits:burned`). Structured as a PURE mapper (`planProjection`, validated at the boundary via `AnyEvent.safeParse` — a malformed event is skipped, never half-written) + a `ProjectionStore` PORT (mirroring ADR-025's JournalStore) with a `Mem` impl (tests + the replay VERIFY, runs in CI without a DB) and a `PrismaRedis` impl (prod). IDEMPOTENT BY event_id: a `ProcessedEvent` inbox row (new Prisma model + migration) is inserted in the SAME `$transaction` as the projected rows — a re-consumed event hits the PK, the whole apply is skipped (exactly-once DB; a P2002 → skip; an aborted tx rolls back the inbox row too, so redelivery is safe). `pos:` is an absolute SET (idempotent); `bal:`/`lb:`/`burned` are INCR deltas applied once via the inbox gate. (3) LEDGER MODEL — `CreditLedger.delta` changed Int→Float (LMSR credits are fractional play-money; Int rejected every real write — a prod poison-loop). Per ADR-026 M1 a BURN is SUPPLY accounting, not a user-balance change (the burned fee is already inside the debit): burns are EXCLUDED from CreditLedger and from bal/lb, and routed to a `credits:burned` supply counter — so `SUM(CreditLedger.delta)` per user == the user's balance == `bal:` (Postgres CreditLedger is the ADR-003 balance AUTHORITY; Redis bal:/lb: are a derived READ cache). (4) LAW — a read-model service writing Redis/Postgres directly is the SANCTIONED pattern (README rule 3 "Redis is hot state … cache derives from events, never the reverse"; worker-ingest's fixtures-hash precedent), not a bus-only-law violation; ALL projection INPUT arrives via `bus.consume`. Play-money invariant intact (a burn is never credited to any balance). Consequences / VERIFY (Mem store, CI-safe; gate 9/9, 120 api tests): replay + 50 scripted orders through a real engine → the projected Position rows EQUAL the engine's per-user positions (and Redis `pos:` agrees) — the "DB == engine state" check; re-consuming the SAME events writes ZERO duplicate rows across every table (inbox dedup); `SUM(CreditLedger.delta)` per user == the `bal:` hot value (no burn drift). Plus a pure emit-mapper test (every published Envelope is a valid typed `AnyEvent`, event_ids are unique + deterministic + valid uuids, outcome index → H/D/A). (−) Deferred follow-ups (surfaced by the idempotency/determinism adversarial review — prod-hardening; several belong to the NEXT phase, chain-auth, which provisions users): (a) FK PARENTS — Order/Fill/Position/CreditLedger have required FKs to User/Market/Match and nothing seeds them yet, so the PrismaRedis path would FK-violate + poison-loop in prod until the auth phase provisions User (on first-seen) and the market lifecycle provisions Market/Match — the Mem VERIFY does not exercise FKs; (b) POSITION SEQ-GUARD — the absolute-set Position has no monotonic guard, so >1 concurrent consumer could apply a stale qty; v1 runs a SINGLE projection consumer (like the engine's single-writer lease) — a source_seq-guarded upsert is required before scaling the group; (c) BAL/LB REBUILD — a crash in the DB-commit→Redis-INCR window drops a bal/lb delta (pos: self-heals on the next position event; bal:/lb: do not) — a rebuild-from-CreditLedger routine is required (Postgres is authoritative); the store comment now states this honestly (no false "rebuild exists" claim); (d) ORDER↔FILL FK ORDERING — orders.v1/fills.v1 are separate topics with independent cursors, so a Fill can precede its Order → FK violation recovered only via xautoclaim reclaim; fix = single ordered topic or an upsert-stub; (e) a DB-backed PrismaRedis integration test (the Mem store structurally can't exhibit ordering/crash hazards); (f) reject-effect + lifecycle-effect bus routing (ADR-028); (g) emit effect outbox for at-least-once (ADR-025). Cross-refs: ADR-003 (off-chain credit ledger = the balance authority these rows feed), ADR-020 (bus planes — the projection consumes the domain plane), ADR-025 (the JournalStore port pattern + the emit/outbox follow-up), ADR-026 (the order command whose effects are projected; the M1 burn rule), README rule 3 (Redis-as-read-model authorization).

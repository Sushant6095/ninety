# ADR 034 chain I/O — tx builders (settle/post_root/claim/mint) with priority fee + confirm-retry, and a Helius webhook → chain_events → settled envelope (packages/chain, apps/api/http/routes/webhooks.ts, prompt 23)

Status: accepted 2026-07-08. Context: prompt 23 — the off-chain plumbing that (a) BUILDS the four omnipitch_core transactions with priority-fee estimation + confirm-and-retry, exports the IDL surface, and (b) INGESTS on-chain settlements via a Helius webhook into the `chain_events` table and re-publishes them as a `settled` bus envelope. Decision (implemented, gate 9/9/9, chain 5 + api-webhook 1 tests): (1) TX BUILDERS (`packages/chain/src/builders.ts`, `@solana/web3.js`-only) — instruction DATA encoders `settleData`/`postRootData`/`claimData`/`mintData` = the anchor discriminator (`sha256("global:<ix>")[:8]`) + borsh-LE args; these exported encoders + the discriminators ARE the IDL surface a caller uses (account metas come from `../pdas.ts`). `priorityFeeIxs(microLamports, cuLimit)` prepends ComputeBudget set-limit + set-price; `estimatePriorityFee` reads recent prioritization fees (75th pct, floored, never throws); `sendWithRetry` confirms and, on failure, refreshes the blockhash and re-sends (bounded) — covering the blockhash-expiry retry path. (2) HELIUS WEBHOOK — `packages/chain/src/helius.ts` is the PURE core: `verifyHeliusSecret` (fail-closed, timing-safe), `parseSettleEvents` (finds our program's `settle_market` instructions in the enhanced-tx batch by discriminator, decodes the result byte + market account), and `processHeliusWebhook(txs, programId, sinks)` which writes EVERY settle to `chain_events` and publishes a `settled` envelope per RESOLVABLE match (sinks injected → unit-tested with no DB/bus). `apps/api/src/http/routes/webhooks.ts` wires the sinks: `POST /webhooks/helius` verifies the shared secret, upserts `ChainEvent` (idempotent by `sig @id` — a Helius retry never duplicates), resolves the market account → matchId, and publishes the settled envelope (source=`chain`, a DETERMINISTIC event_id derived from the tx sig so a redelivery dedups downstream) to the settlement topic. Consequences / VERIFY: (+) tests green — settle parse (ours vs other programs vs non-settle), fail-closed secret, `processHeliusWebhook` (both txs land in chain_events, only the resolvable one publishes, unknown market skipped), builder discriminators + arg encoding + ComputeBudget target; the API route rejects a bad/missing secret (401) and accepts a valid one (empty batch → no DB writes, 200). Gate 9/9/9, api 132 tests. (~) The prompt's headline VERIFY — "a FIRED DEVNET settle appears in chain_events within seconds via the webhook" — requires a real devnet settle transaction, which needs prompt 19 (the txoracle-CPI `settle_market`, still blocked). So the webhook→chain_events→bus PATH is verified end-to-end with a representative mock Helius payload here; wiring a real devnet Helius webhook + firing a devnet settle lands once 19 does. The prompt's stated alternative — a POLLING FALLBACK — is documented: poll `getSignaturesForAddress(programId)` on an interval and feed the same `processHeliusWebhook` (the parse/dispatch is transport-agnostic), so a Helius outage or missing webhook config degrades gracefully. (−) Follow-ups: an idempotent settled-envelope publish that survives a chain_events-written-but-publish-failed crash (an outbox, mirroring ADR-025); the market-account→matchId resolver assumes `Market.id` is the on-chain PDA (confirm when markets are created on-chain in prompt 19); HELIUS_WEBHOOK_SECRET + OMNIPITCH_PROGRAM_ID are prod env. Cross-refs: ADR-030 (the build/IDL that defines the discriminators), ADR-017 (txoracle), prompt 19 (the settle that fires this webhook), prompt 24 (the settlement saga that consumes the settled envelope this publishes).

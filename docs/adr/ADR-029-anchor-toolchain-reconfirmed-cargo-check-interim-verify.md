# ADR 029 anchor build config fixed (program-id, overflow-checks) → `cargo check` compiles; `build-sbf` deploy still blocked (ADR-017 re-confirmed) → cargo-check is the interim on-chain VERIFY (programs/omnipitch_core)

Status: accepted 2026-07-08 (chain-auth phase kickoff). Context: the chain-auth phase (prompts 19–25: settle proof, claim_points, moments cNFT, negative-path suite, tx builders, settlement saga, embedded auth) needs the on-chain program to build. NOW.md carried ADR-017's blocker ("platform-tools cargo 1.79 can't parse edition2024 transitive deps"). Re-checking on 2026-07-08 with the installed toolchain (system cargo 1.95.0, anchor-cli 0.32.1, solana-cli 2.1.0) surfaced the real, layered picture — and two genuine config bugs that blocked even reaching the compiler. Findings / decision: (1) TWO CONFIG BUGS FIXED (were blocking `anchor build` before any compile): (a) `declare_id!` in lib.rs and `[programs.devnet]` in Anchor.toml both held the literal placeholder `"Fill1nAfterFirstDeploy…"`, which is not a valid 32-byte base58 pubkey → Anchor's Pubkey parse threw `Error: String is the wrong size` (and blocked `anchor keys sync` too, since it reads the bad id first). Fixed by generating a program keypair (`target/deploy/omnipitch_core-keypair.json`, gitignored) and pinning its pubkey `6ps8ao7CVhacnRajvFXWTmkknsRnHfEbWmtQ3nDCdBkj` into both. (b) Anchor 0.30+ requires `overflow-checks = true` in the workspace-root `[profile.release]` (on-chain integer overflow must panic, never wrap — a security requirement); added it (+ `resolver = "2"`). (2) THE REAL BLOCKER IS `build-sbf`, NOT `cargo`: `anchor build` shells out to `cargo build-sbf`, which uses the SOLANA PLATFORM-TOOLS' bundled toolchain — `solana-cargo-build-sbf 2.1.0, platform-tools v1.43, rustc 1.79.0` — NOT the system cargo 1.95. rustc 1.79 cannot parse the `edition2024` feature required by `block-buffer 0.12.1`, pulled transitively via `solana-program 1.18.26 → anchor-lang 0.30.1`. So ADR-017's blocker is RE-CONFIRMED and unchanged for the DEPLOYABLE build: the `.so` cannot be produced here, therefore `anchor deploy`, `anchor test`, localnet/devnet settle, and cNFT mint on devnet all CANNOT run. Devnet is otherwise ready (wallet `~/.config/solana/id.json` funded 1.99 SOL; txoracle interface fully mapped in TXLINE-MAP §3 / packages/txline/txoracle.json). (3) BUT `cargo check`/`cargo test` WITH THE SYSTEM cargo 1.95 COMPILE THE PROGRAM CLEANLY (1.95 handles edition2024). DECISION: for the chain-auth phase, the INTERIM on-chain VERIFY is `cargo check -p omnipitch_core` (compilation + type/borrow correctness) PLUS host `cargo test` for logic extracted into pure functions (predicate derivation H/D/A, FINISHED-status gate, Merkle verification) PLUS proof-auditor review — NOT the deployable `anchor test`. On-chain prompts are implemented deploy-ready and verified this far; their devnet/localnet VERIFY is HONESTLY marked BLOCKED (never faked green) until the toolchain is fixed. Off-chain prompts (auth, Helius webhook → chain_events, settlement saga, tx builders as pure instruction construction) reach FULL vitest VERIFY-green normally. (4) UNBLOCK PATH (recommended, an ops task, time-boxed OUT of this session): align to a newer platform-tools with rustc ≥ 1.85 — either bump `anchor-lang` to 0.32 (matching the 0.32.1 CLI) + `solana-program` to the 2.x line whose `cargo build-sbf` ships a newer rust, or install a newer standalone platform-tools (`--tools-version`), or (last resort) pin the RustCrypto stack (block-buffer/digest/sha2) below edition2024 in Cargo.lock — attempted here and NOT a one-liner (the dep graph forces `block-buffer 0.12.1` via `solana-program 1.18`). Consequences: (+) `anchor build` now reaches the compiler (config bugs gone); the program compiles under system cargo; the program ID is real and committed so a future deploy is one `anchor deploy` away once tools are fixed. (+) A precise, honest verification strategy for the phase — no false greens. (−) The on-chain trust layer cannot be devnet-proven in this environment; the phase's on-chain VERIFY blocks pass to an ops toolchain fix. (~) The program keypair is a secret — kept under the gitignored `target/`, only its public id is committed.

# ADR 037 settlement mechanism corrected — statKeys 1/2 total goals, "final" = game_finalised record (off-chain select), on-chain HARD-DISABLED until finality gate + validateStatV2 (supersedes the ADR-036 sentinel decision)

Status: accepted 2026-07-08. Supersedes: the "impossible-sentinel statKeys" decision in ADR-036 (that ADR's C1/C2/H2/one-shot/no-admin properties still stand; only the statKey/period mechanism is replaced here). Context: the TxLINE admin answered the B1 open question directly, and the answer changed the MECHANISM, not just the numbers. My code had guessed home/away = statKeys 1002/1003 and "final" = period 4. Both wrong.

The corrected mechanism (admin-confirmed):
- **statKey 1 = Participant 1 (home) TOTAL GOALS; statKey 2 = Participant 2 (away) TOTAL GOALS** — the anchored stat-validation tree, distinct from the live SSE Stats map (which carries no goals, ADR-015).
- **"Final" is NOT a period value.** It is the score record whose `Action == "game_finalised"`. Settlement selects that record OFF-CHAIN, takes its `Seq`, and proves statKeys 1,2 from it: `GET /api/scores/stat-validation?fixtureId&seq&statKeys=1,2`.
- **Plan A (CPI into `validateStatV2`) is the sanctioned settle path.** The admin twice declined to publish the low-level Merkle hashing spec (hash fn / leaf encoding / sibling order), directing everyone to the `validateStatV2` instruction via the devnet examples — so manual Merkle verification (the old Plan B) is UNSUPPORTED until they publish the spec. Our code already went CPI → vendor-confirmed correct.

Decision — implemented (cargo check green; full lint/test/build green; txline VERIFY green): 
OFF-CHAIN (`packages/txline`, the only place that may call TxLINE): `TxLineClient.settlementProof(fixtureId)` runs the recipe — fetch the scores snapshot, find the `game_finalised` record, read home/away from `Score.*.Total.Goals` (`goalsFromScore`), derive HOME/DRAW/AWAY (`resultFromGoals`: 1/2/3), and prove statKeys 1,2 from that record's `Seq`; returns null if not yet finalised. `ScoreState.Action` added to the schema; `STAT_KEY_HOME_GOALS=1`/`STAT_KEY_AWAY_GOALS=2` in `statkeys.ts` (the whole-table `STAT_KEY_TABLE_CONFIRMED` stays false — only the goals keys are confirmed). The mock serves a synthetic `game_finalised` record so the CI VERIFY exercises the path (home 2 – away 1 ⇒ HOME; 0–0 ⇒ DRAW; 1–3 ⇒ AWAY).
ON-CHAIN (`programs/omnipitch_core`): statKey constants set to the REAL values (1 and 2); the bogus `PERIOD_FINAL` gate removed. The stat-identity requires (`stat_home.key==1`, `stat_away.key==2`) still defend against stat swap/substitution. **Settlement is HARD-DISABLED** by `require!(SETTLEMENT_LIVE, OmniError::SettlementDisabled)` as the first statement of the handler (`SETTLEMENT_LIVE=false`), so every settle reverts before any write.

Why still disabled — the on-chain FINALITY forge (the core unresolved gate): `validate_stat` proves a stat is Merkle-anchored in SOME event sub-tree of the fixture (any match moment), NOT that the proven record is `game_finalised`. The `game_finalised` selection is off-chain. So a permissionless caller could submit a GENUINE proof for statKey 1 taken from a MID-MATCH batch where the home side happened to lead, and settle `result=HOME` even though the match ended a draw or away win — a wrong-result forge via batch/seq selection. Enforcing "this proof IS the finalised record" on-chain needs the sanctioned `validateStatV2` instruction, whose interface/accounts are NOT in our (stale) IDL. Until that exists, the honest state is: correct statKeys wired, off-chain recipe implemented and tested, but the trustless finality gate absent → hard-disabled. This is genuinely fail-closed (an explicit revert, not a sentinel trick — with the real keys now known, the old `u32::MAX` sentinel would have been a lie).

Consequences / VERIFY: (+) `settlementProof` is unit-verified against the mock's `game_finalised` record; result derivation + null-when-not-finalised covered; the on-chain change is cargo-check green with settlement provably unreachable. (~) The devnet real-settle VERIFY remains BLOCKED — needs `validateStatV2`'s interface, a live finished-fixture proof, and TxODDS devnet up (see STEP-0). NOT faked. (−) Going LIVE requires, in order: (1) the `validateStatV2` discriminator + args + accounts from the updated IDL, re-point the CPI; (2) an on-chain finality gate that binds the proof to the `game_finalised` record; (3) the penalty-shootout decision stat (total goals read level for a shootout win → wrong DRAW); then flip `SETTLEMENT_LIVE=true`.

STEP-0 open items to send the admin / resolve (superset of ADR-036's, refined by the admin's answers):
1. **validateStatV2 interface (the #1 hard blocker).** Discriminator, args (borsh layout), accounts — the updated IDL / txoracle.json v-next. Our copy has only `validate_stat`; without V2's interface no correct CPI can be built. Confirm the final instruction name.
1b. **validateStatV2 finality binding (the actual gate keeping SETTLEMENT_LIVE=false).** Does V2 itself reject a proof drawn from a NON-finalised batch — i.e. does it cryptographically bind the proof to the `game_finalised` record — or does it only verify anchoring, leaving us to enforce the authoritative `Seq` on-chain? This is the on-chain-enforcement question (distinct from Q2's off-chain data-model question); the mid-match/seq-selection forge is only closed once this is answered.
2. **Q2 — game_finalised uniqueness (cardinality only; the definition is answered).** Exactly one per fixture, or can a VAR correction re-finalise? If replaceable, which `Seq` is authoritative (we now pick the latest in `settlementProof`)?
7. **game_finalised endpoint presence.** Is the `game_finalised` record returned by the SNAPSHOT (S1) endpoint (which `settlementProof` polls), or only in updates (S2) / the live stream (S3)? If it is transient/not retained in the snapshot, `settlementProof` returns null for every finished fixture and settlement silently never fires.
8. **Devnet anchoring (sharper B3).** Are finished WC26 fixtures' FINAL goal leaves (statKeys 1,2) actually anchored in `daily_scores_roots` on devnet, and does an SL1 devnet subscription return their stat-validation bundles? 4. **Q4 — penalty shootouts (knockout-critical; WC is in knockouts NOW).** statKeys 1,2 are total goals, so a shootout win reads level (e.g. 1–1 won 4–2 on pens) → `home−away` returns DRAW. Does the `game_finalised` record expose a separate winner/decision stat, or do 1,2 somehow encode the decisive result?

# ADR 011 redisbus delivery semantics

Status: accepted 2026-07-07. Context: ADR-007 chose Redis Streams as the day-1 bus driver behind the `Bus` interface but left concrete delivery semantics as a stub; this session implemented `packages/bus/src/redis.ts` fully and had to pin them down. Decision: (1) publish = XADD with approximate trim `MAXLEN ~ 100000` (bounds stream memory to ~100k newest entries); the message key (match_id) is stored in the entry for ordering-aware partitioning. (2) consume = XGROUP CREATE at id "0" (MKSTREAM), so a freshly-created group replays the retained stream and events published BEFORE a consumer joins are still delivered (BUSYGROUP is treated as "already exists"), then a loop: reclaim → `XREADGROUP GROUP g c BLOCK 5000 COUNT 10 STREAMS topic ">"` → handler → XACK. (3) crashed-consumer recovery = XAUTOCLAIM with a configurable min-idle (default 60000ms); each bus instance gets a unique consumer name (`hostname-pid-rand`), so XAUTOCLAIM is the sole path that moves a dead consumer's delivered-but-unacked entries to a live one, running once per loop iteration before the blocking read. (4) semantics = at-least-once: a throwing/crashing handler leaves the entry unacked (redelivered later via XAUTOCLAIM); an unparseable or tombstoned entry is XACKed and dropped so it can never poison-loop the group. (5) interface extension (backward-compatible): `Bus.consume` gained an optional `ConsumeOptions` param (consumer, blockMs, count, minIdleMs) and the `Bus` interface gained `close()` for graceful shutdown — `KafkaBus` gained a no-op `close()` and still throws on publish/consume, its zero-arg methods still structurally satisfying the widened interface. (6) graceful shutdown = `close()` sets a closing flag, disconnects the blocking connections to interrupt any in-progress XREADGROUP, and awaits the loops bounded by a 2s race so a hung handler cannot block shutdown; publish and each consume loop use their own ioredis connection (duplicate) with `maxRetriesPerRequest: null` so blocking reads don't fail as timed-out requests. Consequences: (+) ordering per stream is preserved (XREADGROUP returns entries in id order, handled sequentially); at-least-once with automatic crashed-consumer recovery, verified by an integration test against docker redis (publish 3 → consume in order → ack; clean restart → zero redelivery; kill mid-handle → redelivered exactly once via XAUTOCLAIM). (−) group-create-at-"0" means a brand-new group on a stream with history replays up to ~100k entries — acceptable because groups are created once and durably track their position thereafter; a consumer restart resumes from the group's last-acked id, it does not replay. (−) the integration test requires a reachable redis; it skips (does not fail) when redis is unreachable so `pnpm test` stays green without infra. (−) packages/bus gained a vitest devDep. No change to the bus wire format, topic names, or any CLAUDE.md law or hook; this refines but does not supersede ADR-007.

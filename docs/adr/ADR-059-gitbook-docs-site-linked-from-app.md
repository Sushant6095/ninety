# ADR 059 A GitBook-hosted documentation site is the deep written reference, authored programmatically from repo content and linked from the app

Status: accepted 2026-07-15 (`merge/live-integration`). Cross-ref: ADR-047 (README as submission front door — every doc claim traces to a file/test; the nine pages source ONLY from existing repo content), ADR-051 (the two-source rule the TxLINE page carries verbatim), ADR-036/037 (the settlement forge finding + fail-closed the Settlement page leads with), ADR-015/016 (TxLINE verified live — the endpoint reality these pages document), ADR-056 (landing owns `/` — the header the Docs link lives in), ADR-050 (AppShell — the terminal sub-nav the Docs link also lives in).

Context: judges (and sponsors) read a submission from its docs. The in-app `/how-it-works` is a *visual* proof-flow explainer (React Flow) and the README is the front door, but there was no deep, browsable, written reference. A hosted GitBook site fills that gap — the deep read that complements, not duplicates, the in-app visual. Hard constraint (ADR-047 principle, binding here): no invented facts — every page sources from existing repo content (README, `docs/TXLINE-MAP.md`, ADR-036/037, `docs/BUILD-LOG.md`, CLAUDE.md laws).

Decision:
- **The site is live and canonical for the written reference.** Public URL `https://sushi-2.gitbook.io/ninety-docs/` (org "sushi" KyVeB4VSpDjlEodwya1n · space XaMZcgKQHutGB7v4Qm9G · site site_1owLu · basename `ninety-docs` · free/basic · public). Nine pages: Introduction · The 90-second loop · Getting started · How pricing works · TxLINE integration · Settlement & proofs · Architecture · Play-money & fair play · FAQ. **TxLINE integration** and **Settlement & proofs** are the sponsor-facing depth pages: the former carries the full endpoint map (A1–A3 auth, F1 fixtures, S1–S4 scores, O1–O3 odds, S5 validateStatV2, K1 statKeys), the endpoint→surface map, wire shapes, freshness (SL1 vs SL12) and the two-source rule (ADR-051); the latter leads with the settlement forge finding (ADR-036/037) and the fail-closed decision (`SETTLEMENT_LIVE=false`).
- **Publish mechanics are the reusable REST recipe (GitBook has no API to set up Git Sync — that step is UI-only).** Content was authored fully programmatically against `api.gitbook.com/v1`: create space → open a change request → push pages via `insert_page` ops (≤50 per CR, batched) → **merge the CR** (works on a non-Git-synced space; applies content to the live space) → create the site with `spaces:[spaceId]` → `PATCH` the site `{published:true}` → read back `urls.published`. Dark theme + Ninety-green accent (`#2BD97C`, with a `#12A150` light-mode variant for contrast) applied via a **read-modify-write PUT** to the site customization endpoint. Auto page slugs matched the intended cross-page links (`settlement-and-proofs`, `how-pricing-works`, …).
- **The app links out; it does not embed.** `DOCS_URL = "https://sushi-2.gitbook.io/ninety-docs/"` in `apps/web/src/lib/routes.ts`; a "Docs" item in the landing header (beside "How it works") and in the app-wide terminal sub-nav, both external anchors (`target=_blank rel=noopener`). **`/how-it-works` (React Flow proof-flow) stays the in-app *visual* explainer; GitBook is the deep *written* reference — they complement, never duplicate.**
- **The MCP path is a dead end this session; the REST path is the source of truth.** The `gitbook` MCP server was added at project scope but its tools never appeared in the session registry (a CLI `mcp add` does not hot-load tools) — the raw REST API was driven instead (token authenticates as SUSHANT JAIN / sj317772@gmail.com). Vendored skill packs (`npx skills add` — mattpocock/skills + GitBookIO/gitbook-skills, ~18 dirs under `.agents/skills/`) are **gitignored** to keep them local and out of the submission repo.

Consequences / VERIFY (per the Verification law — verify against the live site, not a local build): the nine pages render on the public URL with the dark theme + green accent; steppers, all three hint styles, tabs and the Architecture mermaid diagram all render (screenshotted and read out loud — endpoints, statKeys, ADR references and program id on the TxLINE/Settlement pages match the repo, no invented facts); clicking "Docs" from `ninety-nu.vercel.app` opens the published site in a new tab. Binding going forward: docs content stays traceable to a repo file/ADR (ADR-047) — a GitBook claim without a repo source is a bug; `/how-it-works` and the GitBook site must not converge into duplicates. Adding a second feed's endpoints or any moving field to the TxLINE page would violate ADR-051. This is a docs/positioning decision — no service, engine, bus, or on-chain behaviour moves (the only code touched is `routes.ts` + the two nav lists in `apps/web`).

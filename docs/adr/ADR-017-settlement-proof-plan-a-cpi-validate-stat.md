# ADR 017 settlement proof — Plan A (CPI txoracle.validate_stat), verified live on devnet

Status: accepted 2026-07-07. Numbering: this is the decision the 07 spike prompt called "ADR-009"; that number is already taken (ADR-009 dev Postgres host port 5433) and ADRs are immutable, so it is filed as the next free number, ADR-017. Context: the 07 proof spike (the highest technical risk in the build) had to decide how `settle_market` verifies a TxLINE score proof on-chain — Plan A = CPI into `txoracle.validate_stat`, or Plan B = replicate TxLINE's Merkle verification inside our `proof.rs` against the `daily_scores_roots` PDA. It required 05 and 06 to be VERIFY GREEN (they are — ADR-015/016). Decision (verified live 2026-07-07): PLAN A (CPI). A real `validate_stat` transaction on devnet (sig `4JRA6qA6h4sZ1XBrrxngBoFXZ3MYRU2h38jVBqU1VKHgfPZfz1RWrpfZ2fBkeGHmS4E3UE1RvtUqvT8tGYUopBZ1`) proved a two-stat predicate (home − away vs 0) against the on-chain PDA `["daily_scores_roots", epochDay u16-LE]`, with `returnData = valid(true)`, consuming only 282,851 CU of the 1.4M budget (1.4M is also the per-tx max). Key facts observed live: `validate_stat` takes ONE read-only account (`daily_scores_merkle_roots`), NO signer, and returns the validity bool via Solana `return_data` — it is NOT an Anchor view (the client reports "Method does not support views", but `.rpc()` + `get_return_data` works). So `settle_market` will CPI `validate_stat` with the single read-only PDA (ComputeBudget request 1.4M), require `get_return_data() == 1`, and only then write the result — 283K CU leaves ample room for `settle_market`'s own logic inside one tx. Plan B is REJECTED: replicating TxLINE's audited Merkle verification in `proof.rs` duplicates trust-critical code for no benefit now that CPI is confirmed feasible and cheap; the verification surface stays in their program. Open items / caveats (named honestly): (1) the scratch-program CPI DEPLOY that would confirm the third-party CPI end-to-end could NOT be built here — platform-tools cargo 1.79 fails to parse transitive deps requiring Rust `edition2024` (`block-buffer` 0.12.1, `zeroize` 1.9.0, …); UNBLOCK = newer Solana platform-tools (cargo ≥ 1.85) or a pinned pre-`edition2024` `Cargo.lock`. This is an environment/toolchain limitation, not a Plan-A risk — the direct devnet call plus Solana's standard invoke/`get_return_data` semantics (1 read-only account, no signer, 283K CU) make CPI feasibility certain; a deploy would only be confirmatory. The same toolchain issue also blocks `anchor build`. (2) STEP 0 partial: statKeys 1002/1003 are NOT final-goals (live: 1002=2, 1003=1 while the score was 1–3), so the final-goals statKey and the FINISHED status field remain ⚠ Day-0 — 07 wiring STEP 0 must resolve them before real settlement. The spike proved the MECHANISM (any two-stat predicate validates on-chain), not the goals mapping. Consequences: (+) settlement can be trustless via CPI with no admin path and low CU; `proof.rs` plan text updated to Plan A. (+) 07's remaining work is wiring (gate FINISHED, build the ix, require `return == 1`), not a research risk. (−) end-to-end CPI deploy and `anchor build` are toolchain-blocked here (named unblock above). (−) the goals statKey and the FINISHED field are still open. No CLAUDE.md law or hook changed — the "there is NO admin result path / the Anchor program verifies TxLINE proofs on-chain" law is upheld and reinforced (CPI keeps verification in TxLINE's program), and the "only `packages/txline` may call TxLINE / only `packages/chain` may build Solana txs" laws stand. Cross-refs ADR-003 (points off-chain / roots on-chain), ADR-015 and ADR-016 (live data spine), TXLINE-MAP §3.

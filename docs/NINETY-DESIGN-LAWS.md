# NINETY — Design Laws

> Ninety is a **premium live football trading exchange** (play-money, WC26). This document is our
> design constitution. It does **not** describe SofaScore — it extracts the timeless principles
> behind world-class live-data interaction and rewrites them as laws for *our* product and *our*
> identity. Evidence for each principle lives in [`sofascore-research/`](sofascore-research/).
>
> These laws sit **above** feature work and beside the existing `CLAUDE.md` design law and the
> `ui-craft` skill. Where they overlap, they reinforce. Violations are bugs.

---

## Design brief (the north star)

```
Direction: Dark-first live-trading terminal — calm, dense, editorial-precise.
           One signature: the Momentum River.
Density:   Compact. Rows for the many, cards for the one. 4px base, 12px default gap.
Surface:   Panels on near-black; depth via surface steps + hairlines. No gradients, no glass.
Type mood: Tight, technical, mono-numeric. Archivo for the hero number, Inter for UI.
Motion:    Crisp ease-out, 50–250ms, asymmetric (enter > exit). Flash = information, never bounce.
Do:        Spend color only on meaning · flash the price · reserve boldness for the River ·
           keep the exact number always visible · one row primitive everywhere.
Don't:     Decorate · animate the numbers' first paint · add a second loud surface ·
           drop focus states · let live updates shift layout · use raw hex in a component.
```

**The convergence test (run before shipping any surface):** *If someone claimed this screen was
generated by a generic template, would you believe them instantly?* If yes, it isn't done.

## I. Foundation

### Law 1 — Quiet everywhere so you can be loud where it counts
The entire product is deliberately restrained — near-monochrome shell, three font weights, flat
surfaces — **so that the one thing that matters explodes off the screen.** For us that one thing is
the **live price** and the **Momentum River**. Everything else earns its calm.
*This is `ui-craft`'s subtract-then-elevate, made law. Boldness is a budget; spend it once.*

### Law 2 — Color is information, never decoration
No color appears unless it *means* something. Chrome is monochrome (one accent + an ink ramp at
descending opacities). Every saturated color is a semantic token: `up #2BD97C`, `down #FF3D81`,
`halt #FFB020`, `chain #9D6BFF`, live-red for in-play.
- **Raw hex in a component is a bug.** Colors come only from `tokens.css`.
- Build **data-encoding ramps**, not flat accents: intensity of a move → intensity of the color, the
  way a great rating scale climbs grey→red→gold→green→cyan→blue. A big up-move is a *stronger* green.
- No decorative gradients, no glass. (Already law; restated because it's load-bearing.)

### Law 3 — Spatial & semantic color constancy
Pick a color for each recurring duality and **never swap it, anywhere, ever**: buy/long = green,
sell/short = pink, halt = amber, on-chain = violet. Once learned, it's read pre-cognitively across
the tape, book, River, and positions. Inconsistency here silently taxes every glance.

### Law 4 — Type earns hierarchy from weight, opacity, and position — not size
Cap the type scale low (a data terminal needs no 64px headline). Build hierarchy from **3 weights
max**, **one-ink-at-several-opacities**, and **placement**. Reserve genuinely large type for the
**one hero number: the price.** (Archivo for the price/hero numerals; Inter for UI.)
- **Tabular numerals on every price, size, time, and stat.** Columns must not jitter when values
  change. One decimal on prices (`61.4`). Non-negotiable.

### Law 5 — Depth via surface steps + hairlines, not shadows
`bg #0B0D10 → surface #14171C → elevated`, edged by the `#232A33` hairline. At most one low-opacity
shadow. A panel reads as lifted because of its surface and edge, not a drop shadow. Reserve a subtle
glow for the **River alone** — it's the sanctioned spotlight.

### Law 6 — Two scales: spacing for gaps, sizes for dimensions
Keep gap/padding tokens separate from component-dimension tokens so row *heights* stay stable when
padding is tuned. Base unit **4px**, default gap **12px** — one notch tighter than a marketing site,
because a trading surface is a data surface. Let hairlines do the separating whitespace would.

## II. Data & Live

### Law 7 — The number is the truth; the visual is the fast read
Every chart, bar, badge, and the River **always carries its exact number** (label or tooltip). The
visual gives instant magnitude; the number gives certainty. Never make a user infer a price from a
picture.

### Law 8 — Live state is first-class and never shifts layout
- **Skeletons reserve exact dimensions** for the book, tape, chart, and positions; content
  cross-fades in (~300ms). A live exchange that jumps is broken (protect CLS).
- **Flash on change** (up-green / down-pink, ~180ms, then settle) — the only motion the numbers get.
- **Live pulses, never spinners**, for in-play state.
- Announce live changes via **polite live regions** so assistive tech hears them without focus theft.

### Law 9 — Charts are system components, in one visual language
Small inline viz (P&L sparklines, implied-probability bars, position heatmaps) render as **SVG**,
themed by tokens, animated within the duration scale — crisp at any DPI, one language across the app.
The heavy price chart uses `lightweight-charts`; the **Momentum River** owns the heavy choreography
(GSAP, per ADR-052). Same green=up/pink=down everywhere. No chart looks like a dropped-in library.

## III. Structure & Navigation

### Law 10 — One skeleton, re-parameterized
Every screen is the same shell (header · detail canvas · live rail) filled with different data.
**Build one row primitive** and reuse it across the tape, market list, leaderboard, and positions;
one table, one card, one badge. Consistency = zero learning cost. A user who learns one screen has
learned them all.

### Law 11 — Depth lives in tabs, not in sprawl
Hold enormous per-entity depth behind **shallow tabs (≤2 levels)**, never behind navigation to new
pages. A market page is *one* URL: `Trade · Order book · Trades · Chart · Rules`. You drill *into* an
entity; you rarely leave it. Reflect the active tab in the URL so any state is linkable.

### Law 12 — Everything is a link; the graph is dense
Every crest, team, player, market, and trader name navigates to that entity. Build the lateral graph
so users never dead-end and rarely need the back button. From a market: both teams, the competition,
the leaderboard, related markets — all one click away.

### Law 13 — Search is the front door
Permanent, central, wide, **keyboard-first** global search across matches · markets · teams ·
players · traders. Grouped, crest-rich results; a non-empty idle state (trending markets, your
watchlist); full combobox semantics. It's the fastest path into the exchange — give it real estate.

### Law 14 — Declare the z-index contract
A single named stacking order (`sticky 97–99 · chrome 100–107 · overlays 108–111` + one escape
hatch), defined in tokens. **No `z-index: 9999` guesses.** This is what makes layered sticky headers
(market header → tab bar → column header) and stacked overlays reliable.

## IV. Motion

### Law 15 — Motion is feedback, bounded and asymmetric
- Everything animates inside **50–250ms** (our tokens; ceiling 250 for UI, longer only for the
  River). Pull *every* animation from the named duration scale — audit that both Framer Motion and
  GSAP obey it (ADR-052).
- **Ease-out, never bounce.** A trading product must feel exact; elastic reads as toy.
- **Enter > exit** (entry more expressive, exit quick).
- **Never animate the numbers' first paint** — information must not wait for a transition.
- Respect `prefers-reduced-motion`: flashes/pulses degrade to instant color changes.

## V. Interaction & Accessibility

### Law 16 — Every interactive element has hover, focus-visible, and active states
No exceptions. The focus ring is **never** suppressed. Buy/sell buttons get the boldest (still ≤150ms,
ease-out) press feedback and airtight `disabled`/`loading` states (halted, insufficient credits,
order pending). A trading UI that drops focus states is a bug.

### Law 17 — Accessibility is load-bearing, not a pass
- **Never color-only.** The number/label always carries the meaning; color reinforces (a colorblind
  user still reads `61.4 ▲`).
- Correct semantics: `tablist`/`tab`/`tabpanel`, `combobox`+`listbox`, `role="switch"`, labeled icon
  buttons, ordered lists for feeds.
- Full keyboard operation; focus trapped in overlays and returned on close; live regions for live
  data. Meets our WCAG-minded bar before ship.

### Law 18 — Least-interruptive tool for the job
Inline popover/tab > drawer > modal. Modals are rare and reserved for genuinely blocking tasks.
Confirm irreversible/large trades with a lightweight inline confirm or sheet (keep flow fast); use
non-blocking **toasts** for fills, halts, and market open/close (right severity, auto-dismiss
non-critical, persist errors).

## VI. Identity — what makes this Ninety, not a scores clone

### Law 19 — Convert engagement into conviction (the core divergence)
A scores product asks "who do you think wins?" and takes a tap; the crowd-% is a poll. **Ninety asks
the same question and answers with a price.** The vote buttons become **buy/sell on outcome
markets**; the crowd bar becomes the **live implied probability from the order book**; "one vote"
becomes a **position you can size, add to, and exit.** Same human instinct, real conviction. Keep the
poll's one-tap low friction for a user's *first* trade — then let the market do the rest.

### Law 20 — The Momentum River is the one loud thing
All visual boldness lives in the **River** (per `CLAUDE.md` and ADR-051/052). It is the hero, the
spotlight surface, the home of heavy motion. Do **not** introduce a second loud surface — a product
with two heroes has none. Everything around the River obeys Laws 1–18 and stays quiet.

### Law 21 — Trade in the language of a market, and keep the play-money covenant
Copy voice: plain verbs, sentence case. **Never** the words *bet / stake / odds / wager* in product
copy — say **price, trade, credits** (existing law, restated as identity). And the play-money
invariant is absolute: no deposits, no cash payouts, ever. Our credibility as a *premium market* and
our integrity as a *play-money* product are the same asset — protect both in every surface.

### Law 22 — WC26 two-source truth (movement vs stillness)
What **moves** during a match (scores, goals, halts, marks, prices, results, who advances) comes from
**TxLINE**; what **sits still** (flags, team metadata, groups, the bracket skeleton, stadiums) comes
from the baked worldcup26 snapshot. Tie-breaker: if a field changes mid-match → TxLINE. Design so the
*moving* data is where the eye and the motion go, and the *still* data is quiet context. (ADR-051.)

## Acceptance checklist (per surface, before "done")

- [ ] **Convergence test** passed — it would not be mistaken for a generic template.
- [ ] Boldness spent only on the price/River; everything else quiet (Laws 1, 20).
- [ ] No raw hex; all color from `tokens.css`; color is semantic (Law 2).
- [ ] Duality colors match the app-wide constants (Law 3).
- [ ] ≤3 weights; hierarchy from weight/opacity/position; tabular numerals; 1-decimal prices (Laws 4).
- [ ] Depth from surface + hairline; no gradient/glass; ≤1 low-opacity shadow (Law 5).
- [ ] Exact number visible alongside every visual (Law 7).
- [ ] Skeletons reserve dimensions; no layout shift; price flashes on change; polite live regions (Law 8).
- [ ] Inline viz is SVG, themed, one visual language (Law 9).
- [ ] Reuses the shared row/table/card/badge primitives (Law 10).
- [ ] Depth in tabs (≤2 levels), active tab in URL (Law 11).
- [ ] Entities are links; no dead ends (Law 12).
- [ ] All motion from the duration scale, ease-out, enter>exit, reduced-motion honored (Law 15).
- [ ] Hover + focus-visible + active on everything; focus ring intact; disabled/loading states (Law 16).
- [ ] Not color-only; correct ARIA roles; keyboard-complete; focus management (Law 17).
- [ ] Least-interruptive overlay chosen; large/irreversible trades confirmed (Law 18).
- [ ] Copy uses price/trade/credits — never bet/stake/odds/wager (Law 21).
- [ ] Moving data (TxLINE) is where motion/attention goes; still data is quiet (Law 22).
